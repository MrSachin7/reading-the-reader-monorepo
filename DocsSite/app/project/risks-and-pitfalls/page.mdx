# Risks and Pitfalls

This page summarizes the main project risks identified during the repo research and planning work.

## Highest-Risk Failure Modes

### AI-first scope drift

The main thesis risk is turning the platform into an AI thesis instead of a modular experiment-platform thesis.

The safer framing is:

- manual
- rule-based
- external-provider seam

not:

- in-repo production AI as the core thesis claim

### Overgrown realtime orchestration

The strongest architectural risk is letting the backend session orchestrator keep absorbing more responsibilities instead of sharpening module seams.

### Realtime coupling across actors

Participant flow, researcher monitoring, derived analytics, and control traffic can become too intertwined if they all behave like one failure domain.

### Weak reproducibility and provenance

Replayable data is not automatically the same thing as defendable experimental provenance. Export metadata, configuration stamping, and provider/intervention provenance matter.

### Privacy and security deferred because the system is local

Participant data, gaze-linked data, and exports still need deliberate handling even in a thesis environment.

## Lower-Level Practical Risks

- contract drift between backend C# shapes and frontend TypeScript mirrors
- hidden hardware assumptions between Windows/Tobii study environments and non-Windows dev environments
- repo-local runtime data becoming operational truth
- intervention behavior causing more disruption than benefit
- broad platformization before the thesis core is fully defendable

## Why This Page Exists

These risks are important because they tell future contributors what not to optimize for.

The next useful changes should improve:

- modularity
- runtime trustworthiness
- reproducibility
- contributor clarity

not just feature count.
